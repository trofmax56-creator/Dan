# Пост: n8n vs ИИ-агент — архитектурный выбор

---

Один из самых частых вопросов от клиентов: «У нас уже есть n8n. Зачем нам ИИ-агент?»

Честный ответ — смотрите на тип задачи.

---

**n8n побеждает, если:**
- Процесс жёсткий и повторяемый: «получил заявку → создал лид → отправил письмо»
- Условия заранее известны и не меняются
- Нужна надёжность: шаблон отработал 10 000 раз без ошибок

**ИИ-агент побеждает, если:**
- Условия меняются: каждый лид приходит с разными данными
- Нужна интерпретация: «квалифицируй лид и реши, кому передать»
- Интеграций много: агент сам решает, какой инструмент вызвать

---

**Главная разница — куда уходит время:**

n8n: новый сценарий = новая схема. Каждый нестандартный кейс — отдельный workflow.

ИИ-агент: первая настройка занимает 3–5 дней (CLAUDE.md, MCP, память). После — агент адаптируется к новым кейсам сам. Gemini CLI показал 156 итераций на одной задаче, пока не «понял» правильный подход — это нормально, это обучение.

---

**Мой вывод после года практики:**

n8n — для операционных рельсов. Агент — для принятия решений.

Идеальный стек: **n8n как Telegram-мост + ИИ-агент как мозг + Bitrix24/MойСклад как данные.**

n8n принимает сообщение → передаёт агенту → агент работает с CRM → возвращает ответ через n8n.

Так вы не выбираете между ними — используете оба по назначению.
