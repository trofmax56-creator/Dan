## Суть
Разработка четырёхуровневой системы защиты для AI-агентов на LangGraph от prompt injection и несанкционированных действий. Система включает валидацию входных данных, контроль инструментов, ограничение действий и аудит операций.

## Бизнес-сценарий
Разработчики AI-агентов на LangGraph защищают систему от prompt injection атак и несанкционированного использования инструментов. Система валидирует запросы пользователей, контролирует какие tools может использовать агент, ограничивает параметры действий и логирует все операции для аудита.

## Алгоритм реализации
1. Шаг 1: Инициализация LangGraph агента с определением инструментов (tools) и параметров выполнения
2. Шаг 2: Реализация первого слоя защиты (Layer 1) — валидация входных данных: проверка типов, длины, формата user input перед передачей в агент
3. Шаг 3: Реализация второго слоя защиты (Layer 2) — контроль инструментов: создание allowlist разрешённых tools, блокировка недопустимых инструментов, проверка наличия инструмента в whitelist перед вызовом
4. Шаг 4: Реализация третьего слоя защиты (Layer 3) — ограничение параметров и действий: валидация аргументов tools, ограничение допустимых значений параметров, проверка области действия операций
5. Шаг 5: Реализация четвёртого слоя защиты (Layer 4) — аудит и логирование: отслеживание всех вызовов инструментов, сохранение истории операций, алерты при подозрительной активности
6. Шаг 6: Интеграция всех четырёх слоёв в единую цепочку обработки запроса с обработкой ошибок и исключений на каждом уровне

## Технический стек
- LangGraph — фреймворк для разработки AI-агентов
- LangChain — для управления цепочками инструментов и обработки данных
- Python — язык программирования для реализации логики защиты
- Pydantic — для валидации схем данных и параметров
- JSON — формат для структурирования параметров инструментов
- Logging модуль Python — для логирования операций
- Type hints (typing module) — для типизации параметров

## Связки инструментов
- User Input → Input Validation Layer 1 → Tool Allowlist Layer 2 → Parameter Validation Layer 3 → Logging & Audit Layer 4 → LangGraph Agent Execution
- LangGraph Agent → Tool Selection → Parameter Extraction → Layer 2 Check (Tool allowed?) → Layer 3 Check (Parameters valid?) → Layer 4 Log (Record action) → Tool Execution

## Конфигурация и параметры
- Layer 1 — Input Validation: проверка max_length строки (например 5000 символов), проверка допустимых символов, проверка формата (например JSON, email, URL)
- Layer 2 — Tool Allowlist: создание списка разрешённых инструментов (allowed_tools = ['read_file', 'send_email']), проверка: if tool_name not in allowed_tools: raise ToolNotAllowed()
- Layer 3 — Parameter Validation: определение schema для каждого инструмента (например max_file_size=1MB, allowed_domains=['example.com']), использование Pydantic models для валидации
- Layer 4 — Audit Logging: создание log entry с timestamp, user_id, tool_name, parameters, result, status; сохранение в БД или файл; определение severity уровней (INFO, WARNING, ERROR)
- Error Handling: Try-Catch блоки на каждом слое, graceful degradation вместо крахов, return error messages вместо stack traces к пользователю
- Rate Limiting: ограничение количества запросов per user per time window (например 100 запросов в минуту)
- Token Limit: ограничение максимального количества токенов в запросе и ответе агента

## Ключевые инсайты
- Prompt injection — наиболее распространённая атака на AI-агентов, когда злоумышленник внедряет вредоносные инструкции в input пользователя; защита от неё — первая линия обороны
- Tool allowlist (белый список) эффективнее чем blacklist, т.к. явно разрешает только необходимые инструменты вместо попытки блокировки всех опасных
- Параметры инструментов требуют отдельной валидации — агент может вызвать разрешённый tool но с опасными аргументами (например delete_file с path='/root')
- Логирование должно быть полным и неизменяемым — для forensics анализа при инцидентах и соответствия compliance требованиям
- Rate limiting предотвращает DoS атаки и контролирует затраты на API вызовы (особенно при использовании платных LLM API)
- Multi-layer defense принцип (defense in depth) означает что даже если один слой пропустит атаку, остальные слои её перехватят
- Sensitive data в логах должно быть masked или encrypted, например passwords, API keys, PII (personal identifiable information)
- User context важен для аудита — логировать не только что произошло но и кто это сделал (user_id, session_id, IP address)

## Подводные камни
- Обходы валидации: злоумышленник может использовать encoding (base64, unicode escape sequences) для обхода input validation — нужна нормализация перед проверкой
- Time-of-check-time-of-use (TOCTOU) race condition: инструмент может быть в allowlist на момент проверки но удалён перед выполнением — используй atomic операции
- Агент может генерировать code который обходит валидацию через рефлексию или динамическое создание функций — тщательная code review необходима
- Performance penalty от дополнительных проверок может быть значительной — balance между security и speed, рассмотри caching результатов валидации
- Логирование sensitive data может нарушить privacy (GDPR, CCPA) — нужна соответствующая обработка и retention policies
- Версионирование схем инструментов: при обновлении параметров tool может произойти несовместимость — используй версионирование в API
- Недостаточное тестирование security: типичные баги — забывчивость проверок, неполные условия, edge cases которые не покрыты в тестах
