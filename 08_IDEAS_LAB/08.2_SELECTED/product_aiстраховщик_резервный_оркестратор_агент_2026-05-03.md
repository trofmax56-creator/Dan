## 1. Название
**AI-Страховщик** — Резервный оркестратор агентов

## 2. Суть
Система из 2–3 взаимозаменяемых ИИ-агентов (Claude + Gemini + резервная модель), где при отказе или исчерпании лимитов одного агента задача автоматически передаётся следующему — бизнес-процессы не останавливаются никогда.

## 3. Бизнес-сценарий
ДО: Компания автоматизировала приём заявок через Claude API. В пятницу вечером у Claude исчерпались лимиты токенов или случился сбой на стороне Anthropic. 50 входящих заявок за выходные не обработаны, менеджеры узнают об этом в понедельник утром. Руководитель говорит: 'Мы снова потеряли заявки из-за того, что ИИ лёг.' ИИ делает: Webhook получает заявку → n8n пробует Claude → если ошибка 429 или timeout за 10 секунд → автоматически переключает на Gemini API → если и Gemini недоступен → задача ставится в очередь SQLite и выполняется при восстановлении. Руководитель получает Telegram-уведомление: 'Claude недоступен с 18:32. Активирован резервный агент Gemini. Обработано 12 заявок без остановки.' Клиент получает: автоматизация работает 24/7 независимо от сбоев конкретного ИИ-провайдера.

## 4. Логика
```
Входящий триггер (Webhook / Wazzup / Bitrix24) → n8n Router → попытка Claude API → при ошибке/таймауте → Gemini API → при ошибке → очередь SQLite + уведомление → Telegram-алерт + метрики
```

## 5. Технический стек
- n8n
- claude-sonnet-4-5
- Gemini API (google-generativeai)

- SQLite (n8n встроенный)

## 6. Связки инструментов
```
Wazzup/Bitrix24 Webhook → n8n Switch-нода (try Claude → fallback Gemini → fallback Queue) → целевой процесс (квалификация / ответ / занесение в CRM) → Telegram алерт при переключении
```

## 7. Алгоритм реализации
1. Создать в n8n универсальный 'агентский роутер': нода Try Claude отправляет запрос на claude-sonnet с таймаутом 10 секунд, при HTTP 429 / 529 / timeout активируется ветка Fallback
2. В ветке Fallback: идентичный запрос отправляется на Gemini 1.5 Flash через официальный Google AI API (ключ без VPN, RF-доступ доступен); промпты унифицированы — один и тот же системный промпт работает на обоих моделях
3. При недоступности обоих агентов: задача сохраняется в SQLite-таблицу pending_tasks (поля: id, payload, created_at, attempts); отдельный cron раз в 15 минут проверяет доступность Claude и перезапускает очередь
4. Настроить Telegram-алерт руководителю: при каждом переключении агента отправлять сообщение с указанием: причина сбоя, время переключения, количество задач в очереди, какой агент сейчас активен
5. Добавить дашборд в Bitrix24 (CRM-аналитика) или Google Sheets: uptime каждого агента по дням, количество fallback-переключений в месяц, задержка обработки при переключении

## 8. Почему купят (ROI)
Час простоя автоматизированного отдела продаж в пиковое время = 5–15 потерянных заявок × средний чек. Для компании с чеком 30 000 руб. — это 150 000–450 000 руб. упущенной выручки за один инцидент. Claude ложится в среднем 2–4 раза в месяц на 30–180 минут. AI-Страховщик делает автоматизацию необслуживаемой. Крючок: 'Один агент — это риск. Система агентов — это бизнес.'
