## Суть
Видео показывает, как подключить AI-агентов к приватным корпоративным данным с помощью RAG (Retrieval-Augmented Generation). Задача: превратить обычного AI-помощника, который только угадывает, в специализированного агента, который работает с конкретными данными компании для точных ответов и автоматизированных действий.

## Бизнес-сценарий
Разработчики и бизнес-автоматизаторы, которые хотят создать AI-агентов для работы с приватными данными компании. Типичные сценарии: автоматизация поддержки клиентов на основе внутренней базы знаний, обработка бизнес-документов, интеграция с CRM/ERP системами, создание чат-ботов, которые отвечают на основе реальных данных, а не общей информации из обучающего набора.

## Алгоритм реализации
1. 1. Определить источники приватных данных (документы, API, базы данных, облачные хранилища)
2. 2. Выбрать и настроить платформу для оркестрации AI-агентов (n8n, Make.com, Zapier или Python + LangChain)
3. 3. Реализовать слой RAG: настроить индексирование документов и поиск релевантной информации
4. 4. Подключить LLM (Claude, GPT-4, Gemini) с контекстом из RAG
5. 5. Интегрировать AI-агента с бизнес-системами через API (Bitrix24, amoCRM, 1C и т.д.)
6. 6. Настроить webhook для входящих событий (письма, чаты, запросы)
7. 7. Реализовать цепочку обработки: webhook → RAG поиск → LLM обработка → действие в системе
8. 8. Протестировать и оптимизировать промпты для контекстного ответа на основе найденных документов

## Конфигурация и параметры
- Выбор источника данных: локальные документы, облачное хранилище (Google Drive, OneDrive), API источники
- Настройка embedding модели: OpenAI text-embedding-3-small или другие в зависимости от LLM
- Размер chunk документа: обычно 1000-1500 символов с overlap 20%
- Параметры RAG поиска: top_k (обычно 3-5), similarity threshold (0.7-0.8)
- Система промптирования: system prompt с инструкциями использовать только найденные данные
- Настройка temperature в LLM: 0.2-0.5 для точности, выше для креативности
- Формат输出 данных: JSON для структурированных ответов
- Обработка контекста: максимальный размер контекста окна, обрезка длинных результатов поиска
- Обработка ошибок: fallback когда нет релевантных документов, сообщение пользователю
- Аутентификация: API ключи для LLM, токены для интеграций, oauth для облачных сервисов

## Ключевые инсайты
- RAG решает главную проблему: AI теперь отвечает только на основе найденных документов компании, а не из обучающего набора
- Для работающего RAG критична хорошая предварительная обработка документов: правильная сегментация на chunks, качественные embeddings
- n8n + Claude / GPT-4 + Pinecone = минимальный стек для production RAG агента, не требует кодирования
- Контекстное окно LLM имеет лимит (например, Claude 100k): нужно отправлять только топ-K релевантных документов, а не все
- Типичная стоимость RAG решения: embedding запросов (~$0.00002 за 1K токенов), поиск в Pinecone (~$0.04 в месяц за 100k размерность), LLM запросы (~$0.003 за 1K входных + $0.015 за 1K выходных токенов для Claude)
- Скорость ответа критична: поиск в векторной БД + LLM inference = 2-5 секунд, нужно кэшировать популярные запросы
- Fallback стратегия важна: если поиск вернул низкий score, лучше ответить 'не знаю' чем галлюцинировать
- Итеративное улучшение: отслеживать какие вопросы были плохо обработаны, добавлять в документы базу, переиндексировать

## Подводные камни
- Старые embeddings несовместимы с новыми моделями: если переходишь с ada-002 на text-embedding-3, нужно переиндексировать весь корпус
- Hallucination всё ещё происходит если LLM дополняет информацию за пределами найденного контекста — нужны строгие инструкции в системном промпте
- Дорогое потребление токенов: долгие документы в контексте = большой input count, нужно минимизировать контекст
- Временная актуальность: документы в RAG могут быть устаревшими, нужна периодическая переиндексация
- Чувствительность к пунктуации и форматированию: PDF с картинками требует OCR (Tesseract), иначе текст не распарсится
- Языковые модели: embeddings обучены в основном на английском, на русском качество может быть ниже — нужны multilingual модели
- Rate limting в API: если много одновременных запросов RAG + LLM, можно получить 429 Too Many Requests
- Размер вектора варьируется: ada-002 = 1536 размерность, text-embedding-3-large = 3072, нужно выбрать под железо и точность
