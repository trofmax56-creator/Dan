## Суть
Автоматизация синхронизации контактов из SmartLead в Attio CRM с автоматическим обновлением категории "Wrong person" при определённых условиях. Workflow извлекает данные о невалидных контактах и обновляет их статус в CRM без ручного вмешательства.

## Бизнес-сценарий
Sales-команды и SDR, работающие со SmartLead (инструмент для холодных писем), нуждаются в автоматическом перемещении невалидных/неправильных контактов в категорию "Wrong person" в Attio CRM. Это экономит время на ручное управление лидами и предотвращает повторные попытки связи с неправильными персонами. Обрабатываются данные о статусах контактов из SmartLead и обновляются соответствующие поля в Attio.

## Алгоритм реализации
1. Шаг 1: Настройка триггера в Make.com — создание Webhook для приёма данных от SmartLead (статус контакта, email, имя, название компании). Webhook URL генерируется автоматически при создании триггера.
2. Шаг 2: Получение данных из SmartLead через Webhook — в payload приходят идентификаторы контактов, их статусы, email-адреса и другая информация о причине пометки как 'Wrong person'.
3. Шаг 3: Поиск контакта в Attio по email или другому уникальному идентификатору (используется модуль 'Search' Attio) для определения record ID контакта, который нужно обновить.
4. Шаг 4: Проверка условия — если статус в SmartLead соответствует 'Wrong person' (может быть несколько кодов: 'invalid_email', 'person_not_found', 'rejected' и т.п.), то переходим к следующему шагу, иначе пропускаем обновление.
5. Шаг 5: Обновление контакта в Attio — модуль 'Update a person' или аналогичный, в котором выставляется поле категории контакта на значение 'Wrong person' (может быть отдельное поле, может быть статус, может быть tag).
6. Шаг 6: Логирование результата — опционально отправка уведомления в Slack или логирование в Google Sheets для аудита, какие контакты были перекатегоризированы и когда.

## Конфигурация и параметры
- SmartLead Webhook: генерируется URL вида https://hook.make.com/... и конфигурируется в SmartLead для отправки событий при изменении статуса контакта на 'Wrong person'
- Attio API Key: вставляется в модуль подключения Attio для авторизации запросов (Settings > API)
- Поле для поиска в Attio: выбирается email как уникальный идентификатор (или можно использовать phone/custom_id)
- Поле обновления в Attio: выбирается поле типа 'Select' или 'Relationship' под названием 'Category' или 'Person Status' с значением 'Wrong person'
- Фильтр условия: проверка переменной из payload: {{trigger.body.status}} === 'wrong_person' или {{trigger.body.reason}} contains 'invalid'
- Rate limiting: Make.com имеет лимиты запросов — рекомендуется добавить sleep 0.5-1 сек между запросами если обновляется много контактов одновременно

## Ключевые инсайты
- SmartLead отправляет webhook в реальном времени при каждом изменении статуса контакта, что позволяет синхронизировать Attio почти мгновенно без задержек
- Для массовых обновлений (если в SmartLead сразу много 'Wrong person') используйте массовые операции Attio вместо одиночных Update запросов — это экономит API calls в 10-20 раз
- Поле 'Wrong person' в Attio может быть не просто статусом, а отдельным relationship или multi-select полем — проверьте структуру custom fields в Attio перед настройкой
- SmartLead webhook может отправлять одновременно несколько типов событий (bounce, unsubscribe, wrong_person) — используйте условие для фильтрации только нужного события, иначе будут ложные обновления
- Если контакт не найден в Attio по email (может быть опечатка в SmartLead или контакт создан вручную с другим форматом), используйте fuzzy search или альтернативный идентификатор (company + full name)
- Для отладки: включите логирование всех запросов в Make.com (History tab) и проверьте, какой ровно payload приходит от SmartLead и какие значения полей в Attio
- Rate limit Attio API — обычно 300 requests/min, при массовых обновлениях это может быть узким местом. Решение: добавить очередь или батч-обработку
- SmartLead может отправлять webhook несколько раз для одного контакта если изменился статус многократно — добавьте дедупликацию по contact_id + timestamp, чтобы не создавать дубли в логах
