# 🗂️ Паспорт идеи: Развёртывание LLM на серверах клиента

**Источник:** [Understanding vLLM with a Hands On Demo](https://youtube.com/watch?v=qdPkA5mxLhg) — KodeKloud (2026-03-31)
**Категория:** Инфраструктура
**Заменяет:** —

---

## 💡 Что это даёт бизнесу
Клиент получает собственный высокопроизводительный LLM-сервер на базе vLLM, не зависящий от OpenAI/Anthropic и их ценообразования. Снижение стоимости инференса в 5–20 раз по сравнению с облачными API при больших объёмах запросов. Данные не покидают инфраструктуру клиента — критично для банков, медицины, госсектора.

---

## ✅ Плюсы
- Независимость от зарубежных API и санкционных рисков — актуально для РФ-рынка
- При объёме от 50 000 запросов/день self-hosted vLLM окупается за 1–2 месяца
- Можно продавать как пакет: железо + настройка + интеграция с CRM/n8n — чек 200–500к

## ❌ Минусы / Риски
- Целевой клиент Студии (МСБ 50–500к бюджет) редко имеет GPU-сервер — нужно убеждать арендовать облачный GPU (RunPod, Vast.ai, Yandex Cloud GPU)
- Высокий порог технической экспертизы: CUDA, Docker, балансировка — Dev=7, легко облажаться на проде

## 🧰 Артефакты для реализации

**Инструменты:** vLLM, Docker, NVIDIA CUDA, Ollama (альтернатива для CPU), n8n, Yandex Cloud GPU
**API:** vLLM OpenAI-compatible REST API, HuggingFace Model Hub API

**Репозитории:**
- github.com/vllm-project/vllm
- github.com/huggingface/text-generation-inference

**Документация:**
- docs.vllm.ai
- huggingface.co/docs/text-generation-inference

## 🚀 Шаги запуска
1. Шаг 1: Определить модель под задачу клиента (Mistral 7B / Llama 3 8B / Qwen2.5) и выбрать GPU-хостинг (RunPod / Yandex Cloud / собственный сервер)
2. Шаг 2: Развернуть vLLM в Docker с OpenAI-совместимым API, настроить continuous batching и quantization (AWQ/GPTQ) для экономии VRAM
3. Шаг 3: Подключить к n8n / Make.com как кастомный HTTP-узел, заменив вызовы OpenAI API на локальный endpoint
4. Шаг 4: Настроить мониторинг (Prometheus + Grafana), автоскейлинг и SLA-документацию, сдать клиенту с инструкцией
