## ЭТАП 0 — СБОР СЫРЬЯ (Data Collection)

### Суть
Автоматический еженедельный мониторинг трёх источников: YouTube, Telegram, статьи. Без ручного поиска.

### Бизнес-сценарий
Конкуренты тратят 3–5 часов в неделю на ручной поиск трендов. Конвейер делает это за 15 минут в фоне, пока команда работает с клиентами.

### Алгоритм реализации

**YouTube (еженедельно, понедельник):**
1. GitHub Actions → запустить `yt_discovery.py` вручную через Run workflow
2. Скрипт ищет видео по ключевым словам: n8n, Claude, CRM, Bitrix24, автоматизация, AI-агент
3. Результат: новые `yt_*.md` со `status: raw` в `00_RAW/YouTube/`
4. Сразу за ним автоматически → `yt_processor_2.py` (в том же workflow)
5. `git pull` локально — получаем обработанные файлы

**Telegram (раз в 2 недели):**
1. Локально: `python 02_TOOLS/Scripts/tg_discovery.py`
2. Читает посты из подписанных каналов за последние N дней
3. Фильтрует по белому списку тематик
4. Сохраняет в `00_RAW/Telegram/`

**Статьи Habr/VC.ru (раз в 2 недели):**
1. Локально: `export FIRECRAWL_API_KEY=fc-... && python 02_TOOLS/Scripts/article_discovery.py`
2. Скрапит по тегам: n8n, CRM, API, Bitrix24, автоматизация
3. Сохраняет в `00_RAW/Articles/`

### Технический стек
- YouTube Data API v3 (бесплатно, лимит 10k/день)
- Telegram MTProto / telethon (сессия `dan_session` в корне проекта)
- Firecrawl API (есть бесплатный план)
- GitHub Actions (Ubuntu latest, Python 3.11)

### Связки инструментов
```
GitHub Actions (cron/manual)
  → yt_discovery.py + YOUTUBE_API_KEY
    → 00_RAW/YouTube/yt_*.md (status: raw)
      → yt_processor_2.py + ANTHROPIC_API_KEY
        → 01_INBOX/Gold/yt_*.md (status: gold)
          → git commit + push (автоматически)
```

### Конфигурация и параметры
```yaml
# .github/workflows/yt_discovery.yml
secrets:
  YOUTUBE_API_KEY: console.cloud.google.com
  ANTHROPIC_API_KEY: console.anthropic.com

# Частота запуска:
  YouTube: еженедельно, понедельник, вручную (workflow_dispatch)
  Telegram: раз в 2 недели, локально
  Articles: раз в 2 недели, локально
```

### Ключевые инсайты
- YouTube блокирует транскрипты с серверов GitHub Actions → скрипт работает без них, классифицирует по заголовку + описанию
- Telegram сессия должна быть активна локально — нельзя запустить в CI
- Firecrawl имеет лимит на бесплатном плане — запускать не чаще раза в 2 недели

## ЭТАП 1 — ОБРАБОТКА И КЛАССИФИКАЦИЯ (Processing)

### Суть
Claude Haiku извлекает структурированную техническую документацию из каждого GOLD-видео. Мусор отсекается автоматически по ключевым словам.

### Бизнес-сценарий
Из 43 сырых видео за 10 минут получаем 40 готовых технических карточек с алгоритмом реализации, стеком и конфигурацией. Аналитик-человек потратил бы на это 2 рабочих дня.

### Алгоритм реализации
1. `yt_processor_2.py` читает все `yt_*.md` со `status: raw`
2. Пробует скачать транскрипт видео (если недоступен — работает без него)
3. Классификация: считает GOLD/TRASH-ключевые слова в тексте
   - GOLD (≥2 совпадения): n8n, Claude, CRM, агент, API, интеграция, кейс, гайд...
   - TRASH (≥2 совпадения): крипта, казино, похудение, реклама, форекс...
4. GOLD → `extract_with_claude()`: отправляет в Claude Haiku с принудительным tool_use
5. Claude возвращает JSON: main_idea, use_case, workflow, tech_stack, tech_links, config_details, key_insights, gotchas, tags
6. Сохраняет в `01_INBOX/Gold/yt_*.md`
7. Обновляет статус в raw-файле: `status: gold` или `status: trash`
8. Фиксирует в `processed_log.md`

### Технический стек
- Claude Haiku (`claude-haiku-4-5`) — быстро и дёшево
- `youtube-transcript-api` — получение субтитров
- Python 3.11, `anthropic` SDK
- GitHub Actions — запуск в облаке

### Связки инструментов
```
yt_*.md (status: raw)
  → fetch_transcript(video_id)  [youtube-transcript-api]
  → classify(text)              [GOLD/TRASH keywords]
  → extract_with_claude(text)   [Claude Haiku tool_use]
  → save_gold()                 [01_INBOX/Gold/yt_*.md]
  → update_raw_file()           [status: gold/trash]
  → update_processed_log()      [processed_log.md]
```

### Конфигурация и параметры
```python
model = "claude-haiku-4-5"
max_tokens = 2048
tool_choice = {"type": "tool", "name": "extract_video_info"}
cache_control = {"type": "ephemeral"}  # экономия токенов на system prompt
стоимость = ~$0.001 за видео
задержка = 3 сек между запросами (защита от блокировки YouTube)
```

### Ключевые инсайты
- `cache_control: ephemeral` на system prompt экономит ~30% токенов при пакетной обработке
- При блокировке YouTube транскриптов — классификация по заголовку+описанию даёт ~85% точности
- Без `ANTHROPIC_API_KEY` скрипт падает в regex-режим: работает, но качество анализа низкое

### 📎 Файлы этапа
| Тип | Файл | Роль |
|---|---|---|
| Скрипт | `02_TOOLS/Scripts/yt_processor_2.py` | Основной скрипт обработки |
| CI/CD | `.github/workflows/yt_discovery.yml` | Запуск в облаке после discovery |
| Вход | `00_RAW/YouTube/yt_*.md` | Сырые файлы со `status: raw` |
| Выход | `01_INBOX/Gold/yt_*.md` | Обработанные GOLD-карточки |
| Лог | `01_INBOX/processed_log.md` | Реестр обработанных ID |

### Артефакты этапа
- `01_INBOX/Gold/yt_*.md` — 407 файлов (68 YouTube + 339 Telegram)
- `01_INBOX/processed_log.md` — лог всех обработанных ID

## ЭТАП 2 — СКОРИНГ ПО DEMAND MATRIX (Selection)

### Суть
Не все GOLD-файлы одинаково ценны. Матрица Pain+Dev+Profit отделяет продукты от просто интересных идей и не даёт тратить ресурсы впустую.

### Бизнес-сценарий
Из 407 GOLD-файлов в продукт реально уйдёт 20–25. Без скоринга команда будет брать "интересное", а не "прибыльное".

### Алгоритм реализации
1. Читаем каждый файл из `01_INBOX/Gold/`
2. Анализируем: теги, tech_stack, key_insights, use_case
3. Выставляем оценки по трём осям:

| Критерий | Вопрос для оценки | Диапазон |
|---|---|---|
| **Pain** | Клиент теряет деньги/время прямо сейчас без этого решения? | 1–10 |
| **Dev** | Можно собрать на n8n + Claude за 1–2 недели без найма? | 1–10 |
| **Profit** | Реально продать за ≥50,000 руб. разово или подписку ≥15,000/мес.? | 1–10 |

4. Сумма определяет маршрут:

```
≥ 25  → 04_PROJECTS          (продукт — берём немедленно)
22-24 → 05_BIZ_RECIPES       (рецепт — упаковываем и продаём)
18-21 → 08_IDEAS_LAB/08.1    (идея — следующий спринт)
контент → 11_1_DRAFTS        (черновик поста)
< 18  → frontmatter: status: archive
```

5. Записываем оценку в frontmatter файла:
```yaml
score: Pain=8 Dev=7 Profit=9 ИТОГ=24
routed_to: 05_BIZ_RECIPES
```

6. Обновляем `08_IDEAS_LAB/Demand_Matrix.md`

### Конфигурация
- Скоринг проводится вручную (Максим + Claude) или через скрипт-автоматизацию
- Периодичность: после каждого нового пакета GOLD-файлов
- Время на 68 YouTube файлов: ~2–3 часа при ручном скоринге

### Ключевые инсайты
- Pain важнее Profit: решения с Pain=9 продаются даже при Dev=5
- Dev < 5 означает нужен внешний разработчик → повышает стоимость, снижает маржу
- Реанимация базы (Reanimator-вектор) всегда имеет Pain ≥ 8 у клиентов с CRM

### 📎 Файлы этапа
| Тип | Файл | Роль |
|---|---|---|
| Матрица | `08_IDEAS_LAB/Demand_Matrix.md` | Таблица оценок Pain+Dev+Profit |
| Вход | `01_INBOX/Gold/` | Все GOLD-файлы для скоринга |
| Выход | frontmatter каждого файла | Поле `score` и `routed_to` |

### Артефакты этапа
- `08_IDEAS_LAB/Demand_Matrix.md` — заполненная матрица по всем файлам
- Файлы с обновлённым frontmatter (score + routed_to)

## ЭТАП 3 — РОУТИНГ ПО ЦЕХАМ ЗАВОДА (Routing)

### Суть
Каждая идея, прошедшая скоринг, попадает в один из 4 цехов и получает конкретного "исполнителя".

### Алгоритм маршрутизации

```
ПО ТЕГАМ И СТЕКУ → ЦЕХ → ПАПКА → ОТВЕТСТВЕННЫЙ

Claude, контент, Reels, статьи, HeyGen
  → Цех Маркетинга → 11_CONTENT FACTORY/11_1_DRAFTS/

Квалификатор, воронка, реанимация базы, AI-Sales
  → Цех Продаж → 04_PROJECTS/AI_Sales_Booster/

n8n, webhook, API, интеграция, схема, автоматизация
  → Цех Сборки → 05_BIZ_RECIPES/ + 04_PROJECTS/

Мониторинг, LTV, поддержка, наставник, обучение
  → Цех Сервиса → 04_PROJECTS/AI_Service/
```

### Структура 4 цехов

**Цех 1: Маркетинг** — Упаковка и смыслы
- Инструменты: Claude API, HeyGen CLI, Midjourney
- Результат: поток лидов через контент

**Цех 2: Продажи** — AI-Sales Booster
- Инструменты: n8n, Bitrix24, Telegram Bot
- Результат: автоматическая квалификация + реанимация базы

**Цех 3: Сборка** — Low-code разработка
- Инструменты: self-hosted n8n, Docker, Claude Code, GitHub Dan
- Результат: работающий ИИ-продукт у клиента за 1–2 недели

**Цех 4: Сервис** — LTV и обучение
- Инструменты: AI-Наставник, мониторинг API, Telegram бот
- Результат: Churn 0%, стабильная подписка

### 📎 Файлы этапа
| Тип | Файл | Роль |
|---|---|---|
| Стратегия | `04_PROJECTS/AI_Factory_Setup.md` | Описание 4 цехов и правила роутинга |
| Вход | Размеченные файлы из ЭТАП 2 | Со score и routed_to |
| Выход | `04_PROJECTS/`, `05_BIZ_RECIPES/`, `08_IDEAS_LAB/`, `11_1_DRAFTS/` | Целевые папки по цеху |

### Артефакты этапа
- Файлы перемещены в целевые папки по цеху
- Обновлён frontmatter: `цех: Сборка`

## ЭТАП 4 — УПАКОВКА ПРОДУКТА (BIZ_RECIPE)

### Суть
Сырая идея из GOLD-файла превращается в готовую карточку продукта по стандартному шаблону. Клиент видит не "интеграцию n8n", а конкретное решение с ценой и сроком.

### Бизнес-сценарий
Карточка BIZ_RECIPE — это одновременно техническое задание для разработчика и коммерческое предложение для клиента. Один документ закрывает обе потребности.

### Алгоритм реализации
1. Берём GOLD-файл с оценкой ≥ 22
2. Определяем вектор по CRM-AI Ideator:
   - **Qualifier** — ускорение обработки входящего лида
   - **Reanimator** — работа с мёртвой базой отказников
   - **Efficiency** — автоматизация рутины (КП, сметы, документы)
3. Создаём файл: `05_BIZ_RECIPES/0XX_название.md`
4. Заполняем по шаблону (ниже)

### Шаблон BIZ_RECIPE
```markdown

title: [AI-Название продукта]
vector: Qualifier / Reanimator / Efficiency

цена_разово: XX,000 руб.
цена_подписка: XX,000 руб./мес.
срок_разработки: X недели
source: yt_VIDEOID.md / tg_POSTID.md
date: YYYY-MM-DD
```

## ЭТАП 6 — МАРКЕТИНГ (Content Factory)

### Суть
Каждый реализованный кейс автоматически превращается в контент по схеме "Якорь + Эхо". Один кейс = 4–5 публикаций на разных площадках.

### Бизнес-сценарий
Контент привлекает входящих лидов без холодных звонков. Технические статьи на Habr формируют экспертный бренд. Посты в Telegram и VC.ru приводят прямых клиентов.

### Алгоритм реализации — схема "Якорь + Эхо"

**Шаг 1: Подготовка материала**
- Берём реализованный кейс из `05_BIZ_RECIPES/` или `04_PROJECTS/`
- Фиксируем: конкретные цифры результата, технические детали, подводные камни

**Шаг 2: Якорь — создаём главный лонгрид**
- Выбираем площадку якоря: VC.ru (бизнес) или Habr (технари)
- Claude генерирует черновик → `11_CONTENT FACTORY/11_1_DRAFTS/`
- Максим правит голос и стиль
- Публикация только после одобрения (правило Dan_Manual)

**Шаг 3: Эхо — адаптируем под площадки**
- Яндекс Дзен → сторителлинг-версия (эмоции, история клиента)
- TenChat → экспертный тезис (3–5 предложений + вывод)
- ВКонтакте → разговорный формат (вопрос к аудитории)
- Telegram канал → короткий инсайт со схемой

**Шаг 4: Видео (опционально)**
- HeyGen CLI генерирует Reels с цифровым аватаром Максима
- Midjourney создаёт обложку / схему для поста

### Площадки и правила

| Площадка | Формат | Цель | Стиль |
|---|---|---|---|
| **Habr** | Технический лонгрид (Якорь) | Экспертный бренд, найм | Код, схемы n8n, API нюансы. Ноль воды |
| **VC.ru** | Бизнес-лонгрид (Якорь) | Привлечение клиентов | Провокационный заголовок, цифры, ROI |
| **Telegram** | Пост + схема | Прогрев аудитории | Инсайт + практика, ссылка на якорь |
| **TenChat** | Экспертный тезис | B2B-нетворкинг | Коротко, вывод, ссылки в комментариях |
| **Яндекс Дзен** | Сторителлинг | SEO + охват | Эмоция, история, короткие абзацы |
| **ВКонтакте** | Разговорный пост | Охват, обсуждение | Вопрос к аудитории, призыв комментировать |

### Технический стек
- Claude API (`claude-sonnet-4-6`) — генерация черновиков постов и адаптация
- HeyGen API — видео-КП и Reels с аватаром
- Midjourney — визуализация схем и обложек
- n8n — автоматическое расписание публикаций

### Связки инструментов
```
Готовый кейс (05_BIZ_RECIPES/)
  → Claude API (генерация черновика якоря)
  → 11_1_DRAFTS/пост_название.md
  → Одобрение Максима
  → Публикация: Habr/VC.ru (якорь)
  → Claude API (адаптация под Дзен/TenChat/ВК)
  → n8n (расписание публикаций)
  → HeyGen CLI (видео из текста кейса)
```

### Конфигурация промпта для генерации контента
```

Пиши в стиле практика, а не теоретика.

Для Habr: добавь код/схемы, API-детали, конфигурацию нод n8n
Для VC.ru: начни с провокации, добавь ROI клиента, призыв в конце
Для Telegram: один инсайт на пост, схема стрелками, не более 800 символов
```

### Ключевые инсайты

- Один технический кейс на Habr приносит лиды 6–12 месяцев после публикации

## ЭТАП 7 — ПРОДАЖИ (Sales Workshop)

### Суть
Клиент видит работающий прототип + кейс с цифрами. Продажа происходит на демо, а не на переговорах о стоимости.

### Бизнес-сценарий
Входящий лид из контента уже "прогрет" — он читал кейс и видел результат. AI-Квалификатор сортирует лидов автоматически, менеджер тратит время только на тех, кто готов платить.

### Алгоритм реализации
1. Лид приходит через контент (Telegram / VC.ru / Habr / сарафан)
2. AI-Квалификатор в Bitrix24 (n8n workflow) определяет тип клиента и потребность
3. Автоматически: создаётся сделка в CRM, назначается менеджер, отправляется первое письмо
4. Менеджер проводит демо на живом прототипе (15–20 мин)
5. Оффер: фикс за внедрение + подписка на поддержку
6. Договор → предоплата 50% → старт разработки

### Ценовая матрица

| Продукт | Внедрение | Поддержка/мес. | Срок |
|---|---|---|---|
| Простая интеграция (1 workflow) | 30–50к руб. | 10к руб. | 1 неделя |
| AI-агент для CRM | 80–150к руб. | 20к руб. | 2 недели |
| Комплексный AI-Sales Booster | 150–250к руб. | 30к руб. | 3–4 недели |
| Завод ИИ-решений под ключ | 300–500к руб. | 50к руб. | 2–3 месяца |

### Технический стек
- Bitrix24 (воронка продаж, CRM)
- n8n (AI-Квалификатор, автоматизация воронки)
- Claude API (анализ запроса лида, генерация КП)
- Telegram Bot (уведомления менеджеру)

### Ключевые инсайты
- Демо на реальных данных клиента закрывает 70% возражений по цене
- Подписка на поддержку важнее разового платежа — строит предсказуемый доход
- AI-Reanimator по базе отказников даёт первые продажи уже в первую неделю

## ЭТАП 8 — СЕРВИС И LTV (Service Workshop)

После внедрения клиент не уходит — AI-Наставник отвечает на вопросы 24/7, мониторинг ловит баги до жалобы клиента.

### Алгоритм реализации
1. При запуске проекта — создаём базу знаний клиента в `03_GUIDES/clients/`
2. AI-Наставник (Telegram бот на Claude) отвечает на вопросы по системе
3. n8n Cron каждые 15 минут проверяет доступность API и ключевых интеграций
4. При ошибке — автоматическое уведомление в Telegram (нашей команде + клиенту)
5. Раз в квартал — аудит системы и предложение нового модуля (upsell)

### Технический стек
- Claude API — AI-Наставник (бот в Telegram)
- n8n Cron — мониторинг API
- Telegram Bot API — уведомления
- `03_GUIDES/clients/` — база знаний клиента

### Конфигурация мониторинга
```
n8n Cron: каждые 15 минут
  → HTTP Request: проверка Bitrix24 API
  → HTTP Request: проверка Claude API
  → IF: ошибка?
    → Telegram: уведомление команде
    → Telegram: уведомление клиенту
```

### Ключевые инсайты
- Проактивный мониторинг устраняет 80% причин отказа от подписки
- AI-Наставник снижает нагрузку на поддержку в 3–4 раза
