## Суть
Сравнение трёх платформ для автоматизации бизнес-процессов (Zapier, Make и n8n) с целью определить оптимальный инструмент для различных типов задач автоматизации в 2026 году.

## Бизнес-сценарий
Предприниматели, операционные менеджеры и разработчики выбирают между Zapier, Make и n8n для автоматизации бизнес-процессов, интеграции CRM, управления данными и рабочих процессов. Анализируются функциональность, стоимость, простота использования и возможности интеграции каждой платформы.

## Алгоритм реализации
1. 1. Определение критериев сравнения: цена, простота использования, количество интеграций, гибкость, скорость выполнения и поддержка
2. 2. Анализ Zapier: архитектура, ценообразование, встроенные интеграции, визуальный конструктор, ограничения на задачи (tasks)
3. 3. Анализ Make: расширенные функции (scenarios), условная логика, производительность, сложные рабочие процессы
4. 4. Анализ n8n: самостоятельное развёртывание (self-hosted), полный контроль, открытый исходный код, технические требования
5. 5. Практическое сравнение: примеры реальных автоматизаций в каждой платформе с демонстрацией интерфейса
6. 6. Вывод: рекомендации по выбору в зависимости от масштаба бизнеса, технических навыков и бюджета

## Технический стек
- Zapier (облачная платформа)
- Make (облачная платформа)
- n8n (облачная и самостоятельная версия)
- REST API
- Webhooks
- Интеграции CRM (Salesforce, HubSpot и др.)
- Интеграции с электронной почтой (Gmail, Outlook)

## Связки инструментов
- Webhook → Zapier/Make/n8n → API интегрируемого сервиса
- Trigger (событие) → Action (действие) → Multiple Actions (множественные действия)
- Conditional Logic (условная логика) → Branching (разветвление рабочего потока)
- Error Handling → Retry Logic → Logging & Monitoring

## Конфигурация и параметры
- Zapier: Task = 1 выполненная автоматизация (1 задача), Premium план включает количество задач в месяц
- Make: Execution = 1 операция в workflow, сценарии с циклами и условиями требуют большего количества operations
- n8n: No operation limits в облачной версии, self-hosted версия — прямое использование сервера
- Webhook URL как источник данных (trigger)
- API ключ для авторизации в целевом сервисе
- Расписание (Schedule trigger) для периодического запуска
- Переменные и data mapping между шагами workflow
- Error handling: retry, skip, stop workflow

## Ключевые инсайты
- Zapier — лидер по количеству готовых интеграций (5000+), идеален для малого бизнеса с простыми автоматизациями
- Make — лучше для средних компаний: позволяет создавать сложные многоуровневые сценарии с условной логикой
- n8n — выбор разработчиков и крупных компаний: полный контроль, возможность self-hosting, модульная архитектура
- Zapier дороже при большом количестве tasks (от $30/мес за базовый план, логарифмический рост стоимости)
- Make имеет более гибкую тарификацию: платите за operations, а не за количество zaps
- n8n облачная версия стартует от $10/мес с лимитом на выполнения, но self-hosted вариант почти бесплатен
- Производительность: n8n fast, Make средний, Zapier может быть медленнее на сложных операциях
- Кривая обучения: Zapier простой, Make средний, n8n крутой (требует знания Node.js/JSON)
- Все три поддерживают Webhooks, REST API, OAuth, но n8n предоставляет больше низкоуровневого контроля
- Надёжность: Zapier — 99.9% uptime, Make — 99.9%, n8n self-hosted — зависит от вашей инфраструктуры

## Подводные камни
- Zapier: Task count vs Actions — confusion между понятиями, неправильный расчёт стоимости
- Zapier: Rate limiting на интеграциях третьих сервисов может замедлить автоматизацию
- Make: Operations counter считает нестандартно (например, 1 loop iteration = 1 operation, может неожиданно увеличить счёт)
- Make: Complexity penalties — очень сложные scenario стоят дороже в плане operations
- n8n self-hosted: требует знания Docker, управления базами данных, резервными копиями и безопасностью
- n8n: меньше готовых интеграций по сравнению с Zapier/Make, требует написания custom code
- Zapier не рекомендуется для высоконагруженных систем (миллионы tasks в месяц)
- Make может быть нестабильным при экстремально больших данных в workflow
- Все платформы требуют тестирования перед production-deployment
- Миграция между платформами может быть дорогой и трудоёмкой — не планируйте частые переходы
