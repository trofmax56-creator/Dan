## Суть
ИИ-агент автоматизирует взаимодействие с клиентами в переписке, выполняя функции менеджера по продажам: выявляет потребности, отвечает на вопросы и проводит продажи без участия человека.

## Бизнес-сценарий
Компании с отделом продаж используют ИИ-агент для автоматизации клиентской переписки (в мессенджерах, письмах): агент общается как живой менеджер, квалифицирует лиды, выявляет потребности и закрывает сделки, что снижает нагрузку на команду и ускоряет цикл продаж.

## Алгоритм реализации
1. 1. Входящее сообщение от клиента поступает в систему (Telegram, WhatsApp, email или веб-форма) через webhook или API интеграцию
2. 2. Триггер активирует workflow в автоматизационной платформе (n8n/Make.com) - проверяется тип сообщения и язык
3. 3. Текст сообщения передается в LLM (Claude/ChatGPT/Gemini) с контекстом: история диалога, информация о клиенте, база знаний о продуктах
4. 4. ИИ-модель анализирует сообщение, определяет намерение (вопрос, возражение, согласие), выявляет потребности клиента
5. 5. LLM генерирует ответ в тоне менеджера, учитывая стратегию продаж: задает уточняющие вопросы, преодолевает возражения, предлагает решение
6. 6. Сгенерированный ответ отправляется обратно клиенту в исходный канал (Telegram/WhatsApp/email) через API интеграцию
7. 7. История переписки логируется в CRM (Bitrix24/amoCRM) с теми/категоризацией статусов (новый лид, квалифицирован, в сделке, закрыто)
8. 8. При обнаружении готовности к покупке или критических вопросов система может подтянуть живого менеджера через уведомление или эскалацию

## Технический стек
- LLM платформы: Claude API, ChatGPT API, Gemini
- Автоматизационные платформы: n8n, Make.com, Zapier
- Интеграция мессенджеров: Telegram Bot API, WhatsApp Business API, Twilio
- CRM системы: Bitrix24, amoCRM, Pipedrive
- Инструменты для работы с контекстом: RAG (Retrieval-Augmented Generation), векторные БД (Pinecone, Weaviate)
- Веб-фреймворки: Node.js, Python (FastAPI, Django)
- Хранилище переписки: PostgreSQL, MongoDB, JSON файлы
- Мониторинг: Webhook, REST API, GraphQL endpoints

## Связки инструментов
- Входящее сообщение (Telegram/WhatsApp) → Webhook → n8n/Make.com
- n8n: узел HTTP Request → узел Code (парсинг сообщения)
- n8n → Claude API (POST запрос с промптом и контекстом)
- Claude API → n8n (обработка ответа)
- n8n → API мессенджера (отправка ответа клиенту)
- n8n → Bitrix24/amoCRM API (логирование переписки, обновление статуса контакта)
- Альтернативный поток эскалации: условие IF → уведомление менеджеру → webhook в Slack/Telegram

## Конфигурация и параметры
- Webhook URL в Telegram Bot API: настройка webhook на адрес n8n/Make.com
- Claude API параметры: model='claude-3-sonnet' (для скорости) или 'claude-3-opus' (для качества), max_tokens=1000, temperature=0.7 (естественность речи)
- Промпт система: прописываются роль ИИ (менеджер по продажам), стиль общения, правила (не обещать невозможное), правила эскалации
- Контекст для LLM: история последних 5-10 сообщений, информация о клиенте (имя, компания, статус), каталог продуктов/услуг
- Интеграция Bitrix24: API endpoint для добавления комментария https://api.bitrix24.com/b24api/v1/batch, поля: UF_CRM_* для кастомных полей
- amoCRM интеграция: REST API для обновления сделки (поле responsible_user_id, статус pipeline_id/status_id)
- Условия эскалации: если клиент упоминает техническую проблему И это не из FAQ → маршрут техподдержку, если сумма > X → маршрут к менеджеру
- Логирование: каждое сообщение сохраняется в БД с timestamp, sender_type (client/ai), confidence_score, intent_label

## Ключевые инсайты
- Для естественного диалога критично передавать LLM полную историю переписки (не менее 5 последних сообщений), иначе ИИ повторяет рекомендации или игнорирует уже полученную информацию
- Temperature параметра LLM должен быть 0.6-0.8: при 0 ответы слишком роботизированные, при 1.0 ИИ может забыть о цели продажи
- Промпт должен включать конкретные шаги квалификации: сначала узнать проблему, потом предложить решение, потом уточнить бюджет/сроки (не наоборот)
- Для сложных объектов продажи (B2B, высокая стоимость) нужна эскалация при достижении определённых вех (рассчитанный ROI, согласие на демо)
- Кэширование промпта и контекста в Claude API экономит токены на 90%: одно и то же введение для каждого клиента передается 1 раз в cache_control
- Если использовать RAG с базой знаний, вероятность галлюцинаций ИИ снижается на 80% — обязательна векторная БД с документами о продуктах/политиках
- Время отклика имеет значение: ИИ-ответ должен генерироваться < 2 сек, иначе клиент теряет интерес; рекомендуется streaming ответов для визуальной быстроты
- Интеграция sentiment-анализа в workflow позволяет выявить раздражённого клиента и автоматически маршрутизировать к живому менеджеру с повышенным приоритетом
