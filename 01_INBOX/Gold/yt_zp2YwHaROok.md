## Связки инструментов
- n8n Node Editor → API интеграции → Внешние сервисы
- Webhook → n8n Trigger → Workflow Logic → HTTP Request / Database Query
- n8n Database Node → Custom Code Node → External API

## Конфигурация и параметры
- Типы нод n8n: HTTP Request, Webhook, Function, Loop, Conditional, Database Query
- Параметры API интеграций: endpoints, authentication tokens, request methods (GET, POST, PUT)
- Настройки workflow: trigger conditions, error handling, retry logic
- Возможные значения полей: API keys, webhook URLs, data mapping configurations

## Ключевые инсайты
- n8n требует значительных вычислительных ресурсов при масштабировании на большое количество автоматизаций
- Проблемы с производительностью возникают при одновременном выполнении множества workflows
- Кривая обучения n8n достаточно крутая для сложных многоуровневых автоматизаций
- Отсутствие встроенной поддержки некоторых специфических интеграций может затянуть разработку
- Альтернативные решения (Zapier, Make) проще для быстрого прототипирования, но дороже на масштабе
- Custom разработка на Python может быть более экономичной для специализированных задач с большим объемом
- Миграция из n8n требует переписывания логики workflows, что занимает время и ресурсы

## Подводные камни
- ⚠️ n8n не всегда оптимален для высоконагруженных систем с тысячами одновременных executions
- ⚠️ Миграция существующих workflows требует полного переписывания, т.к. логика нод отличается между платформами
- ⚠️ Затраты на self-hosted n8n (серверы, поддержка) могут быть выше, чем на облачные аналоги
- ⚠️ Некоторые интеграции в n8n требуют кастомного кода, что усложняет no-code подход
- ⚠️ Версионирование и rollback workflows в n8n требует дополнительных процессов и настроек
- ⚠️ Документирование сложных workflows в n8n может быть затруднено из-за визуального интерфейса
- ⚠️ Цена облачного n8n растёт вместе с количеством executions, что может стать критичным при масштабировании
